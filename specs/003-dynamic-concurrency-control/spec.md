# 功能规格说明书：动态并发控制管理

**功能分支**: `003-dynamic-concurrency-control`  
**创建日期**: 2025-10-22  
**状态**: 草稿  
**输入**: 用户描述: "RobotTaskScheduler 的concurrencySemaphores可以通过接口修改"

## 用户场景与测试

### 用户故事 1 - 运行时调整动作类型并发限制 (优先级: P1)

系统管理员需要在系统运行期间根据实际负载情况动态调整不同动作类型的并发执行数限制，而无需重启应用。例如，在高峰期降低语音任务并发数以避免服务器过载，或在低峰期提高消息任务并发数以加快处理速度。

**设定此优先级的理由**: 这是核心功能，直接实现用户需求——允许动态调整并发控制。没有此功能，管理员无法在运行时优化系统资源分配，必须重启服务才能调整配置。

**独立测试**: 可通过调用管理接口修改特定动作类型的并发限制，并验证新限制立即生效，现有任务继续执行不受影响。

**验收场景**:

1. **假如** 系统当前 SEND_MESSAGE 并发限制为 10，**当** 管理员通过接口将其调整为 15 时，**那么** 系统应立即应用新限制，允许最多 15 个消息任务并发执行
2. **假如** 系统当前 SEND_VOICE 并发限制为 5，**当** 管理员通过接口将其调整为 3 时，**那么** 系统应立即应用新限制，当前正在执行的 5 个任务继续完成，但新任务受 3 个并发限制
3. **假如** 系统正常运行中，**当** 管理员调整任意动作类型的并发限制时，**那么** 其他动作类型的并发限制不受影响，独立生效

---

### 用户故事 2 - 查询当前并发配置状态 (优先级: P2)

系统管理员需要查询当前所有动作类型的并发限制配置和实时使用情况，以便评估系统负载状态并做出调整决策。

**设定此优先级的理由**: 查询接口是调整操作的前提，帮助管理员了解当前状态。虽然重要，但相比核心的修改功能属于辅助功能。

**独立测试**: 可通过调用查询接口获取所有动作类型的并发配置和使用情况，验证返回数据准确反映系统当前状态。

**验收场景**:

1. **假如** 系统已配置 3 种动作类型的并发限制，**当** 管理员查询并发配置时，**那么** 系统应返回所有动作类型的并发限制值、当前使用槽位数、可用槽位数
2. **假如** 某动作类型的并发槽位已全部占用，**当** 管理员查询配置时，**那么** 系统应明确显示该类型的可用槽位为 0，使用率为 100%

---

### 用户故事 3 - 配置修改的持久化和审计 (优先级: P3)

系统管理员希望对并发限制的修改操作进行记录，以便审计和追溯配置变更历史。可选地，修改后的配置能够持久化，在系统重启后保持最新配置。

**设定此优先级的理由**: 审计和持久化是管理规范性功能，有助于运维管理，但不影响核心的动态调整能力。

**独立测试**: 可通过修改并发配置后检查日志或审计记录，验证变更被正确记录。可选验证系统重启后配置是否保持。

**验收场景**:

1. **假如** 管理员修改了某动作类型的并发限制，**当** 修改成功后，**那么** 系统应记录操作日志，包含操作时间、操作人、修改前后的值
2. **假如** 管理员在运行时修改了并发配置，**当** 系统重启后，**那么** 配置可以保持最新修改的值（可选功能，可以优先支持内存修改）

---

### 边缘场景

- 当管理员将并发限制设置为 0 时，系统如何处理？（应拒绝或允许但警告）
- 当管理员设置的并发限制超过线程池容量时，系统如何提示？
- 当并发限制被降低且当前执行任务数超过新限制时，如何处理正在执行的任务？（应允许现有任务完成，但阻止新任务）
- 当多个管理员同时修改同一动作类型的并发限制时，如何处理并发冲突？
- 当系统不存在某动作类型的并发配置时，如何响应查询或修改请求？

## 需求

### 功能性需求

- **FR-001**: 系统**必须**提供 RESTful API 接口，允许授权用户查询所有动作类型的并发限制配置
- **FR-002**: 系统**必须**提供 RESTful API 接口，允许授权用户修改指定动作类型的并发限制值
- **FR-003**: 系统**必须**在接收到修改请求后立即应用新的并发限制，无需重启服务
- **FR-004**: 系统**必须**验证修改请求的合法性：并发限制必须为正整数（大于 0）
- **FR-005**: 系统**必须**在并发限制被降低时，允许当前正在执行的任务完成，但阻止新任务超过新限制
- **FR-006**: 系统**必须**记录每次并发限制修改操作的日志，包含操作时间、动作类型、修改前后的值
- **FR-007**: 系统**必须**在查询接口返回每个动作类型的并发限制值、当前使用槽位数、可用槽位数、使用率
- **FR-008**: 系统**必须**对修改接口进行权限控制，仅允许管理员角色访问
- **FR-009**: 系统**必须**在并发限制修改后更新相关的 Prometheus 监控指标（如已集成监控）
- **FR-010**: 系统**应该**在修改请求时验证新限制是否超过线程池容量，如超过则给出警告

### 关键实体

- **并发配置实体**: 表示某个动作类型的并发控制配置，包含属性：动作类型（SEND_MESSAGE/SEND_VOICE/SEND_NOTIFICATION）、并发限制值、当前使用槽位数、可用槽位数、使用率
- **配置修改记录**: 表示一次并发配置的修改操作，包含属性：操作时间、动作类型、修改前的值、修改后的值、操作人（可选）

## 成功标准

### 可衡量的成果

- **SC-001**: 管理员能在 5 秒内通过接口成功修改任意动作类型的并发限制，新限制立即生效
- **SC-002**: 系统在并发限制修改后，100% 的新任务遵循新限制，0% 的正在执行任务被中断
- **SC-003**: 查询接口的响应时间不超过 1 秒，返回数据准确反映实时状态（误差小于 100 毫秒）
- **SC-004**: 所有并发限制修改操作 100% 被记录到日志中，包含完整的操作信息
- **SC-005**: 并发限制修改接口对非授权用户的拒绝率为 100%（安全性验证）
- **SC-006**: 系统在高负载情况下（1000+ 并发任务）修改并发限制，不影响现有任务的执行成功率（保持 99%+ 成功率）

## 范围边界

### 包含在本功能中

- 提供查询和修改并发限制的 RESTful API 接口
- 动态更新 RobotTaskScheduler 中的 Semaphore 实例
- 基本的参数验证（正整数、范围检查）
- 操作日志记录
- 权限控制（基于现有权限体系）
- 实时并发状态查询

### 不包含在本功能中

- 配置修改的持久化存储（初版仅支持内存修改，重启后恢复默认配置）
- 复杂的审计系统（仅记录日志，不提供审计查询接口）
- 配置修改的回滚功能
- 自动化的并发限制调整（基于负载自动调节）
- 批量修改多个动作类型的并发限制（需逐个修改）

## 假设

- 系统已有基本的权限控制体系，可以区分管理员和普通用户
- RobotTaskScheduler 是单例服务，不存在多实例部署的并发配置同步问题
- 并发限制的修改频率较低（每小时不超过 10 次），不需要考虑频繁修改的性能优化
- 动作类型是预定义的（SEND_MESSAGE、SEND_VOICE、SEND_NOTIFICATION），不支持动态添加新动作类型
- 线程池配置（executorMaxPoolSize）在运行时不会修改
- 管理员了解系统的资源容量，能够合理设置并发限制值

## 依赖

- Spring Boot Web 框架（提供 RESTful API 支持）
- RobotTaskScheduler 服务已正常运行
- RobotTaskConfiguration 配置类（提供默认配置值）
- 现有的权限控制机制（用于接口鉴权）
- Prometheus 监控集成（可选，用于更新监控指标）

## 约束

- 并发限制修改必须是线程安全的，避免并发修改导致的数据不一致
- 不能中断或影响正在执行的任务
- 接口响应时间必须在 1 秒以内，避免影响管理操作体验
- 修改操作必须是原子性的，要么完全成功要么完全失败
- - 系统重启后配置恢复为 application.properties 中的默认值（初版约束）
