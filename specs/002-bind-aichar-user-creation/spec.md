# 功能规格说明书：AI角色与用户账户自动绑定

**功能分支**: `002-bind-aichar-user-creation`  
**创建日期**: 2025-10-21  
**状态**: 草稿  
**输入**: 用户描述: "ai character 创建的同时创建users表的记录, 并绑定"

## 用户场景与测试 *（必填）*

### 用户故事 1 - AI角色创建自动生成用户账户 (优先级: P1)

作为系统管理员或真实用户,当我创建一个AI角色时,系统应自动为该AI角色创建一个对应的用户账户,并将二者关联起来,使得AI角色可以作为一个独立的用户实体参与到系统的社交功能中。

**设定此优先级的理由**: 这是整个AI角色系统的基础功能。AI角色必须拥有用户账户才能参与好友关系、消息交互等社交功能。这是后续所有社交功能的前提条件。

**独立测试**: 可以通过创建一个AI角色,验证系统自动生成了用户账户,并且可以查询到该用户账户的完整信息(用户类型、昵称、用户名等),确认两者正确关联。

**验收场景**:

1. **假如** 我是已登录的真实用户,**当** 我通过API创建一个AI角色(提供角色名称"小助手"和头像URL)时,**那么** 系统应成功返回AI角色ID和关联的用户账户ID
2. **假如** AI角色创建成功,**当** 我查询该AI角色的详细信息时,**那么** 我应能看到关联的用户账户ID(associatedUserId字段)
3. **假如** AI角色创建成功,**当** 我查询用户表中对应的用户记录时,**那么** 该用户记录应存在,且用户类型为"AI",昵称与AI角色名称一致,头像与AI角色头像一致
4. **假如** AI角色创建成功,**当** 我查看自动生成的用户账户时,**那么** 用户名应遵循命名规则(如"ai_character_{角色ID}"),邮箱应遵循规则(如"ai_character_{角色ID}@ai.internal")
5. **假如** 创建AI角色的请求缺少必填字段(如角色名称),**当** 我提交请求时,**那么** 系统应拒绝创建并返回明确的错误信息,且不应创建任何用户账户

---

### 用户故事 2 - AI角色信息与用户账户同步 (优先级: P1)

作为系统管理员,当我更新AI角色的基本信息(如名称或头像)时,系统应自动同步更新关联用户账户的对应信息,确保数据一致性。

**设定此优先级的理由**: 数据一致性是系统稳定性的基础。如果AI角色和用户账户信息不同步,会导致用户界面显示混乱,影响用户体验。此功能与创建功能同等重要。

**独立测试**: 可以先创建一个AI角色,然后更新其名称和头像,验证关联的用户账户的昵称和头像也同步更新。

**验收场景**:

1. **假如** 我已创建一个AI角色,**当** 我更新该AI角色的名称时,**那么** 关联用户账户的昵称应自动更新为新名称
2. **假如** 我已创建一个AI角色,**当** 我更新该AI角色的头像URL时,**那么** 关联用户账户的头像应自动更新为新头像URL
3. **假如** 我同时更新AI角色的名称和头像,**当** 更新完成时,**那么** 关联用户账户的昵称和头像应同时更新
4. **假如** AI角色更新失败(如验证错误),**当** 系统回滚更新操作时,**那么** 关联用户账户的信息应保持不变

---

### 用户故事 3 - AI角色删除自动处理用户账户 (优先级: P2)

作为系统管理员,当我删除一个AI角色时,系统应自动标记关联用户账户为已删除状态,确保不会留下孤儿数据,同时保留历史记录用于审计。

**设定此优先级的理由**: 这是完整的生命周期管理功能,确保数据的完整性和可追溯性。虽然重要,但优先级低于创建和更新功能。

**独立测试**: 可以创建一个AI角色,然后删除它,验证AI角色被标记为已删除,关联的用户账户也被标记为已删除,且历史数据仍可查询。

**验收场景**:

1. **假如** 我是AI角色的创建者,**当** 我删除该AI角色时,**那么** AI角色应被标记为已删除(软删除)
2. **假如** AI角色被成功删除,**当** 我查询关联的用户账户时,**那么** 该用户账户应被标记为已删除(is_deleted = 1)
3. **假如** AI角色已被删除,**当** 其他用户尝试搜索用户时,**那么** 该AI用户不应出现在搜索结果中
4. **假如** AI角色已被删除,**当** 管理员查询历史数据时,**那么** 应能通过特殊查询看到已删除的AI角色和用户记录

---

### 用户故事 4 - AI用户在社交功能中的集成 (优先级: P2)

作为真实用户,我应能够将AI角色关联的用户账户添加为好友,并在好友列表和搜索结果中看到明确的AI用户标识,以区分真实用户和AI用户。

**设定此优先级的理由**: 这是AI角色功能的实际应用场景,使AI角色能够真正融入社交系统。依赖于用户账户的创建和绑定功能。

**独立测试**: 可以创建一个AI角色,然后用真实用户账户搜索该AI用户并添加为好友,验证能够成功添加,且在好友列表中正确显示AI用户类型。

**验收场景**:

1. **假如** 系统中存在公开的AI角色,**当** 我搜索用户时,**那么** AI角色关联的用户账户应出现在搜索结果中,并带有"AI"类型标识
2. **假如** 我找到一个AI用户,**当** 我添加该用户为好友时,**那么** 系统应允许添加,且添加成功后显示在我的好友列表中
3. **假如** AI用户已在我的好友列表中,**当** 我查看好友列表时,**那么** 该用户应显示userType为"AI"的标识
4. **假如** AI用户是私有的(visibility = PRIVATE),**当** 非创建者搜索用户时,**那么** 该AI用户不应出现在搜索结果中

---

### 边缘场景

- 当AI角色创建过程中数据库操作失败时会怎么样？系统应回滚整个事务,确保不会创建孤儿的AI角色或用户账户
- 当用户表已存在相同用户名时(理论上不应该发生),如何处理？系统应在生成用户名时添加额外的唯一标识(如UUID后缀)
- 当AI角色更新请求只包含部分字段时,如何处理用户账户同步？系统应只同步实际更新的字段(名称和头像),其他字段保持不变
- 当系统中存在大量AI角色时,如何确保用户名和邮箱的唯一性？使用AI角色的数据库自增ID作为唯一标识的一部分
- 当AI角色的创建者被删除时,AI角色和关联用户账户如何处理？AI角色应保留,但标记为"孤儿"状态,仍然可以被其他用户使用
- 当尝试手动创建userType为AI的用户账户时,如何防止？系统应只允许通过AI角色创建流程自动创建AI用户,拒绝手动创建

## 需求 *（必填）*

### 功能性需求

**AI角色与用户账户创建**

- **FR-001**: 系统**必须**在创建AI角色时自动创建对应的用户账户
- **FR-002**: 系统**必须**将创建的用户账户的ID存储在AI角色的associatedUserId字段中
- **FR-003**: 系统**必须**将自动创建的用户账户的userType设置为"AI"
- **FR-004**: 系统**必须**将AI角色的名称同步到用户账户的nickName字段
- **FR-005**: 系统**必须**将AI角色的头像URL同步到用户账户的avatar字段
- **FR-006**: 系统**必须**为AI用户生成唯一的用户名,格式为"ai_character_{角色ID}"
- **FR-007**: 系统**必须**为AI用户生成唯一的邮箱,格式为"ai_character_{角色ID}@ai.internal"
- **FR-008**: 系统**必须**为AI用户生成随机密码(AI用户不能通过密码登录,密码仅用于满足数据库约束)
- **FR-009**: 系统**必须**将AI用户的status设置为1(启用状态)
- **FR-010**: 系统**必须**在一个事务中完成AI角色和用户账户的创建,确保数据一致性

**数据同步**

- **FR-011**: 系统**必须**在更新AI角色名称时,同步更新关联用户账户的nickName
- **FR-012**: 系统**必须**在更新AI角色头像时,同步更新关联用户账户的avatar
- **FR-013**: 系统**必须**在一个事务中完成AI角色和用户账户的更新,确保数据一致性
- **FR-014**: 系统**必须**在更新失败时回滚所有更改,保持数据一致性

**删除处理**

- **FR-015**: 系统**必须**在软删除AI角色时,同时标记关联用户账户为已删除(is_deleted = 1)
- **FR-016**: 系统**必须**在一个事务中完成AI角色和用户账户的软删除
- **FR-017**: 系统**必须**保留已删除的AI角色和用户账户数据,用于历史记录和审计

**查询和验证**

- **FR-018**: 系统**必须**在查询AI角色详情时返回associatedUserId字段
- **FR-019**: 系统**必须**允许通过associatedUserId查询对应的AI角色
- **FR-020**: 系统**必须**在用户搜索功能中包含AI用户(当AI角色为PUBLIC时)
- **FR-021**: 系统**必须**在搜索结果和好友列表中显示用户的userType,以区分真实用户和AI用户
- **FR-022**: 系统**必须**防止通过普通用户注册接口创建userType为"AI"的用户

**权限和安全**

- **FR-023**: 系统**必须**禁止AI用户通过密码登录API
- **FR-024**: 系统**必须**只允许AI角色的创建者更新和删除该角色及关联用户
- **FR-025**: 系统**必须**验证创建AI角色的用户是真实用户(userType不是AI)

### 关键实体

- **AiCharacter(AI角色)**: 代表系统中的AI虚拟角色,包含角色的基本信息(名称、头像、可见性等)和关联用户ID(associatedUserId)。关联用户ID指向users表中对应的用户记录。
- **User(用户)**: 代表系统中的用户账户,可以是真实用户或AI用户。通过userType字段区分("REAL"表示真实用户,"AI"表示AI角色关联的用户)。AI用户的账户信息从关联的AI角色同步而来。
- **关联关系**: AI角色与用户账户是一对一关系。每个AI角色有且仅有一个关联的用户账户,通过associatedUserId字段维护关联。

## 成功标准 *（必填）*

### 可衡量的成果

- **SC-001**: 创建AI角色后,系统能在1秒内自动创建并关联用户账户,且关联关系可立即查询验证
- **SC-002**: 更新AI角色名称或头像后,关联用户账户的对应字段能在1秒内同步更新完成
- **SC-003**: 100%的AI角色创建操作要么完全成功(角色和用户都创建),要么完全失败(都不创建),不会出现孤儿数据
- **SC-004**: AI用户能够正常出现在用户搜索结果中(当可见性为PUBLIC时),且在界面上能清晰区分AI用户和真实用户
- **SC-005**: 真实用户能够成功添加AI用户为好友,且添加成功率达到99%以上(排除网络故障等外部因素)
- **SC-006**: 删除AI角色后,关联的AI用户不再出现在用户搜索结果和新的好友添加操作中
- **SC-007**: 系统能够支持至少10000个AI角色及其关联用户账户的并发存在,查询性能不降低
