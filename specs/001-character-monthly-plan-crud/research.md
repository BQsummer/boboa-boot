# 技术研究：虚拟人物月度计划表管理

**功能分支**: `001-character-monthly-plan-crud`  
**研究日期**: 2025-11-28  
**状态**: 已完成

## 研究任务清单

### 1. 日期规则格式设计

**决策**：采用键值对格式存储日期规则

**理由**：
- 格式简洁易懂：`day=5` 表示每月5号，`weekday=1,week=2` 表示每月第二个周一
- 易于解析：可通过简单的字符串分割和键值映射解析
- 扩展性好：未来可添加新的规则类型而不影响现有数据

**曾考虑的备选方案**：
| 方案 | 优点 | 缺点 |
|------|------|------|
| JSON格式 `{"type":"fixed","day":5}` | 结构化好 | 过于复杂，增加解析成本 |
| Cron表达式 `0 0 9 5 * ?` | 标准化 | 对月度规则表达力不足，用户不易理解 |
| 自然语言 "每月5号" | 用户友好 | 需要NLP解析，实现复杂 |

**最终格式定义**：
```
固定日期：day=N          其中 N ∈ [1, 31]
相对日期：weekday=W,week=K   其中 W ∈ [1, 7]（周一到周日），K ∈ [1, 5]（第几周）
```

### 2. 时间字段类型选择

**决策**：使用 MySQL `TIME` 类型存储 `start_time`

**理由**：
- 直接对应 Java 的 `LocalTime` 类型
- 无需日期部分，节省存储空间
- 支持标准时间格式 HH:mm:ss

**曾考虑的备选方案**：
| 方案 | 优点 | 缺点 |
|------|------|------|
| VARCHAR 存储 "09:00" | 简单 | 无法利用数据库时间函数 |
| INT 存储分钟数 540 | 紧凑 | 可读性差，需要转换 |
| DATETIME | MySQL原生 | 包含不需要的日期部分 |

### 3. JSON 字段处理方案

**决策**：使用 MySQL `JSON` 类型 + Java `String` 类型映射

**理由**：
- MySQL 5.7+ 原生支持 JSON 类型
- 项目已有类似用法（如 `prompt_template.tags`）
- 使用 String 类型避免引入额外 JSON 库依赖

**最佳实践**：
- 存储前验证 JSON 格式有效性
- 使用 Jackson 进行 JSON 校验
- API 层返回时直接作为 JSON 对象

### 4. 权限控制策略

**决策**：复用现有 AiCharacter 权限模式

**理由**：
- 月度计划属于 AiCharacter 的子资源
- 权限应继承自 AiCharacter 的创建者
- 保持架构一致性

**实现方式**：
1. 通过 `character_id` 查询 AiCharacter
2. 验证 `created_by_user_id` 是否等于当前用户 ID
3. 不通过则返回 403 权限错误

### 5. 软删除策略

**决策**：使用 `is_deleted` 字段，值为 0/1

**理由**：
- 与现有表结构一致（users、ai_characters 等都使用此模式）
- 便于数据恢复和审计
- 查询时添加 `is_deleted = 0` 条件

### 6. 外键约束决策

**决策**：添加 `character_id` 外键约束到 `ai_characters` 表

**理由**：
- 确保数据完整性
- 级联删除：当 AI 角色被删除时，相关计划也应被删除
- 与现有表结构一致（如 `ai_character_settings`）

**曾考虑的备选方案**：
| 方案 | 优点 | 缺点 |
|------|------|------|
| 无外键，应用层控制 | 灵活 | 可能产生孤儿数据 |
| 软外键（不建实际约束）| 性能好 | 无法保证数据完整性 |

## 技术要点总结

| 项目 | 决策 | 依据 |
|------|------|------|
| 日期规则格式 | `day=N` 或 `weekday=W,week=K` | 简洁、易解析、可扩展 |
| 时间类型 | MySQL TIME / Java LocalTime | 精确匹配，无冗余 |
| JSON 字段 | MySQL JSON / Java String | 项目已有模式 |
| 权限控制 | 基于 AiCharacter.createdByUserId | 复用现有逻辑 |
| 删除策略 | 软删除 (is_deleted) | 项目标准模式 |
| 外键约束 | 添加，级联删除 | 保证数据完整性 |
