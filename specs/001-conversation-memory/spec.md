# Feature Specification: 对话记忆系统

**Feature Branch**: `001-conversation-memory`  
**Created**: 2026-01-23  
**Status**: Draft  
**Input**: User description: "实现对话记忆系统，包括原始消息存储、滚动总结、长期记忆向量检索"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 保持连贯的多轮对话 (Priority: P1)

作为系统用户，我希望在与系统对话时，系统能记住我们之前的所有交流内容，这样我就不需要重复已经说过的信息，对话可以自然地延续下去。

**Why this priority**: 这是对话记忆系统的核心价值。如果系统无法记住对话历史，就无法提供连贯的对话体验，用户将需要不断重复信息。这是最基础也最关键的功能。

**Independent Test**: 可以通过以下方式独立测试：用户在同一会话中发送多条消息，系统的回复需要体现出对之前对话内容的理解。例如，用户说"我养了一只猫"，之后问"它吃什么"，系统应该理解"它"指的是猫。

**Acceptance Scenarios**:

1. **Given** 用户开始一个新对话，**When** 用户连续发送3条相关的消息，**Then** 系统的每条回复都应该体现出对之前所有消息的理解
2. **Given** 用户在对话中提到某个信息（如"我叫张三"），**When** 用户在后续消息中询问相关信息，**Then** 系统应该能够引用之前提到的信息
3. **Given** 对话已经进行了10轮以上，**When** 用户提到很早之前讨论的话题，**Then** 系统应该能够回忆起相关内容并作出恰当回应

---

### User Story 2 - 长对话的智能总结 (Priority: P2)

作为频繁使用系统的用户，我希望即使对话进行了很长时间（数百条消息），系统仍能保持高效响应，不会因为处理大量历史消息而变慢，同时还能准确把握整个对话的脉络。

**Why this priority**: 当对话历史变得很长时，如果每次都加载所有原始消息会影响性能和成本。通过智能总结，可以在保持对话连贯性的同时提升效率。这是基础记忆功能的重要优化。

**Independent Test**: 可以通过模拟长对话测试：让用户与系统进行50轮以上的对话，观察系统在第50轮的响应时间是否与第5轮相近，同时验证系统是否仍能准确引用早期对话的关键信息。

**Acceptance Scenarios**:

1. **Given** 对话已经进行了30轮，**When** 系统需要生成新回复，**Then** 系统应该在合理时间内（不超过正常响应时间的2倍）完成，不因历史消息过多而显著变慢
2. **Given** 对话中包含多个不同主题的讨论，**When** 用户询问"我们之前都聊了什么"，**Then** 系统应该能够提供一个结构化的总结，涵盖主要话题
3. **Given** 对话历史达到一定长度（如30条消息），**When** 系统内部进行总结处理，**Then** 后续对话仍应能准确理解之前的上下文

---

### User Story 3 - 记住重要信息和用户偏好 (Priority: P2)

作为系统的长期用户，我希望系统能够记住关于我的重要信息、我的偏好、我不喜欢的沟通方式，以及我反复提到的重要事件，这样系统就能提供更个性化和贴心的服务。

**Why this priority**: 这让系统从"记住对话"升级为"理解用户"。通过记住用户的长期信息，可以显著提升用户体验和满意度。这是P2而非P1，因为即使没有长期记忆，基本的对话功能仍能运作。

**Independent Test**: 可以独立测试：用户在一次对话中明确表达某个强烈偏好（如"请不要跟我说教"），然后在另一次全新的对话中，观察系统是否记住并遵守这个偏好。

**Acceptance Scenarios**:

1. **Given** 用户在某次对话中明确表达了强烈的情绪或偏好，**When** 系统判断这是需要长期记住的信息，**Then** 系统应该将这个信息保存为长期记忆
2. **Given** 用户开始一个新的对话会话，**When** 讨论的话题与之前某个长期记忆相关，**Then** 系统应该能够调取相关记忆并体现在回复中
3. **Given** 用户反复提到某个重要事件（如失业、失恋），**When** 系统检测到这是高重要性的信息，**Then** 系统应该提升这条记忆的重要性等级，确保未来能优先调用

---

### User Story 4 - 跨会话的智能记忆检索 (Priority: P3)

作为系统用户，我希望当我提到某个话题时，系统能够智能地联想到之前所有相关的讨论和记忆，即使那些对话发生在很久以前或者用的是不同的表达方式。

**Why this priority**: 这是最高级的功能，让系统具备真正的"联想记忆"能力。通过语义检索而非简单的关键词匹配，可以大幅提升记忆召回的准确性。这是P3因为前面的功能已经能支撑基本使用，这是锦上添花的优化。

**Independent Test**: 可以独立测试：用户在第一次对话中讨论"我最近工作压力大"，几天后在新对话中说"我感觉很累"，系统应该能够联想到之前关于工作压力的记忆，即使表达方式不同。

**Acceptance Scenarios**:

1. **Given** 用户过去的对话中包含多条关于某个主题的记忆，**When** 用户在新对话中用相似但不完全相同的词语提到该主题，**Then** 系统应该能够检索到相关的历史记忆
2. **Given** 系统中存储了用户的多条长期记忆，**When** 用户当前的话题可能与多条记忆相关，**Then** 系统应该按照重要性和相关性排序，优先调用最合适的记忆
3. **Given** 某条记忆已经很久未被访问，**When** 系统进行记忆检索时，**Then** 系统应该适当降低该记忆的优先级，避免调用过时或不再相关的信息

---

### Edge Cases

- 当用户在同一会话中前后陈述矛盾的信息时（如先说"我喜欢猫"，后说"我不喜欢猫"），系统应该如何处理？应该记住哪个版本？
- 当对话历史非常大（如超过1000条消息）时，系统如何确保总结的准确性和检索效率？
- 当用户明确要求"忘记我刚才说的话"或"删除某条信息"时，系统应该如何响应？
- 如何处理多个用户同时进行大量对话导致的高并发记忆写入和检索？
- 当向量检索返回的记忆与当前对话关联度较低时，系统应该忽略还是仍然使用？
- 如何处理用户身份切换或多账号场景，确保不会混淆不同用户的记忆？

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统必须能够保存用户与系统之间的每一条原始消息，包括消息的发送者、内容、时间戳等基本信息
- **FR-002**: 系统必须能够识别同一个用户的不同对话会话，并为每个用户维护独立的消息历史
- **FR-003**: 系统必须能够在对话历史达到一定长度时（如30条消息）自动触发总结机制
- **FR-004**: 系统必须能够生成结构化的对话总结，总结应该涵盖对话的关键信息、主题和上下文
- **FR-005**: 系统必须能够记录每次总结所覆盖的消息范围，避免重复总结或遗漏消息
- **FR-006**: 系统必须能够识别对话中的重要信息，包括但不限于：用户的强烈情绪、明确表达的偏好、重要生活事件
- **FR-007**: 系统必须能够为每条长期记忆分配重要性等级，用于后续的优先级排序
- **FR-008**: 系统必须能够将长期记忆的文本内容转换为向量表示，以支持语义检索
- **FR-009**: 系统必须能够为每条长期记忆标记类型（如事件、偏好、关系、情绪等），以便分类管理
- **FR-010**: 系统必须能够在生成回复前，根据当前对话内容检索相关的长期记忆
- **FR-011**: 系统必须能够根据记忆的重要性、相关性和访问时间对检索结果进行排序和过滤
- **FR-012**: 系统必须能够更新记忆的访问时间，用于追踪记忆的使用频率
- **FR-013**: 系统必须能够将检索到的相关记忆整合到回复生成的上下文中
- **FR-014**: 系统必须能够控制总结的触发时机，避免过于频繁或过于稀疏的总结
- **FR-015**: 系统必须能够处理用户身份信息，确保不同用户的记忆互不干扰

### Key Entities

- **原始消息 (Message)**: 表示用户与系统之间的单次交互，包含消息内容、发送者身份（用户或系统）、所属用户ID、消息时间戳、消息唯一标识符。这是对话记忆的最基础单元，记录了所有未经处理的原始对话数据。

- **对话总结 (Summary)**: 表示对一段连续对话的结构化摘要，包含总结内容（以结构化格式存储，如JSON）、所属用户ID、总结覆盖的最后一条消息ID、创建时间戳。总结的目的是压缩长对话历史，同时保留关键信息。

- **长期记忆 (Memory)**: 表示从对话中提取的值得长期保存的重要信息片段，包含记忆文本内容、向量表示（用于语义检索）、所属用户ID、记忆类型（事件/偏好/关系/情绪等）、重要性等级（数值表示）、最后访问时间、创建时间。每条长期记忆都是独立的知识单元，可以跨会话检索和使用。

- **用户 (User)**: 代表系统的使用者，是消息、总结、记忆的归属主体。所有记忆相关的数据都必须关联到具体的用户，确保记忆的隔离性和个性化。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 在包含20轮以上对话的会话中，系统能够准确引用10轮之前的对话内容，准确率达到90%以上
- **SC-002**: 当对话历史超过50条消息时，系统生成回复的响应时间不应超过初始对话（少于10条消息）响应时间的1.5倍
- **SC-003**: 用户在第一次对话中表达的明确偏好或重要信息，在第二次对话中应该有80%以上的概率被系统正确识别并记住
- **SC-004**: 当用户提到与历史记忆相关的话题时，系统检索到的前3条记忆中至少有2条是真正相关的（相关性由人工评估确定）
- **SC-005**: 对于单个用户的对话，系统能够处理至少100轮对话而不出现明显的性能降级或记忆丢失
- **SC-006**: 在多用户并发场景下（至少10个用户同时对话），每个用户的记忆不应出现混淆或错误关联
- **SC-007**: 用户对系统"记忆能力"的主观满意度评分（1-5分）平均达到4分以上

## Assumptions *(assumptions made during specification)*

- 假设系统已经具备基本的对话生成能力，本功能专注于记忆管理
- 假设用户的唯一标识（user_id）已经在系统中可用且稳定
- 假设系统具备调用大型语言模型进行文本总结和记忆提取的能力
- 假设系统具备将文本转换为向量表示（embedding）的能力，可通过调用embedding模型实现
- 假设存在合适的数据存储方案支持消息、总结和记忆的持久化
- 假设向量检索功能可通过集成向量数据库或向量检索服务实现
- 假设单个用户的对话历史在合理时间范围内（如1个月）不会超过10000条消息
- 假设总结触发阈值（如30条消息）是基于性能和成本的合理平衡点，可在后续根据实际情况调整
- 假设长期记忆的数量在单个用户维度不会无限增长，系统可通过重要性和时间衰减机制进行管理
- 假设记忆的重要性评分（0-1或1-5）将由大型语言模型基于对话内容自动判断

## Dependencies *(if applicable)*

- 依赖大型语言模型服务用于对话总结生成
- 依赖大型语言模型服务用于判断是否需要创建长期记忆以及提取记忆内容
- 依赖文本embedding模型服务用于生成记忆的向量表示
- 依赖向量检索能力（向量数据库或检索服务）用于语义相似度搜索
- 依赖用户身份认证系统提供稳定的用户标识
- 依赖数据存储服务（数据库）用于持久化消息、总结和记忆数据

## Out of Scope *(explicitly excluded)*

- 不包括跨用户的记忆共享或群组记忆功能
- 不包括记忆的手动编辑或管理界面（如用户自行删除/修改某条记忆）
- 不包括记忆的导出、备份或迁移功能
- 不包括记忆内容的加密或特殊隐私保护机制（假设使用标准的数据安全措施）
- 不包括多模态记忆（如图片、语音记忆），仅支持文本记忆
- 不包括记忆的自动过期或定时清理机制（当前阶段）
- 不包括基于记忆的主动提醒或推荐功能（如"你上次说想做X，现在做了吗"）
- 不包括记忆分析或统计功能（如显示用户最常讨论的话题）
